# ClipTray: Clipboard History with Hotkey Picker

**Status:** Design approved 2026-05-15. Implementation pending.
**Scope:** New app at `apps/ClipTray/`. New shared type `WindowsTrayCore.HotkeyRegistration` (consumed by ClipTray; ProgramHider migrates onto it in the same phase). No changes to `WindowsAppCore`. Install script + CI workflow gain ClipTray entries.

## Summary

ClipTray is the fifth tray utility in the suite. It captures clipboard text and image content (up to user-configured caps), persists history across restarts, and exposes a hotkey-driven popup picker for fast paste-back. Privacy is layered: skip-sync markers, a pause toggle, a foreground-app blocklist, lock-screen auto-pause, and a length-and-charset password heuristic that flags (not drops) suspected secrets. Captured items dedup by SHA-256; images live as PNG sidecars to keep `index.json` small. The shared `HotkeyRegistration` type lifts hotkey logic out of ProgramHider into `WindowsTrayCore` for reuse.

## Context

Windows ships its own clipboard history (Win+V) which handles text, images, and persistence, but it's anaemic on the parts power users want: no pinning, no search/filter, no regex, no exclude-rules beyond the system-wide "do not sync" marker. ClipTray is the targeted improvement: text plus images (parity with Win+V), plus search, regex, pinning, and a layered privacy model.

ProgramHider already has a local global-hotkey implementation (its show/hide trigger). The Phase 18 retrospective specifically flagged "lift HotkeyRegistration into the shared lib once a second consumer appears." ClipTray is that consumer.

The four existing apps (BatteryTray, NetProfileSwitcher, ProgramHider, SoundTracker) share `WindowsAppCore` (logging, settings, single-instance, startup, update checker) and `WindowsTrayCore` (TrayIcon, TrayTheme, ThemeApplier, TrayTooltipBuilder, AccentColors, VersionFormatter, AboutDialog, StandardMenuItems). ClipTray follows the same composition and adds two new running surfaces: a clipboard-format-change listener (Win32 `AddClipboardFormatListener`) and a global-hotkey registration (Win32 `RegisterHotKey`). Both live behind hidden `NativeWindow`s using the exact pattern already in `TrayIcon` and `TrayTheme`.

## Locked Decisions

| Axis | Decision |
|---|---|
| Content types | Text (CF_UNICODETEXT, CF_TEXT fallback) + Images (CF_DIB / CF_BITMAP). HTML / RTF stripped to text. File lists (CF_HDROP) out of scope. |
| Default caps | 50 text items + 10 image items, both user-configurable. FIFO eviction; pinned items exempt. Default disk quota 100 MB. |
| Trigger | Global hotkey opens picker at cursor (default Ctrl+Shift+V). Tray-menu "Show picker" item for discoverability. |
| Picker UX | Borderless popup, focused search box, virtualised list, pinned items pinned to top, regex when input starts with `/`. |
| Persistence | `%LOCALAPPDATA%\ClipTray\` with `index.json` manifest + `items\<sha256>.png` sidecars. Survives restarts. |
| Privacy layers | All four enabled by default: skip-sync markers, pause toggle, foreground blocklist (seeded with common password managers), lock-screen pause, password heuristic that flags rather than drops. |
| Hotkey shared lib | New `WindowsTrayCore.HotkeyRegistration` type. ProgramHider migrates onto it in the same phase; the app-local hotkey class is deleted. |
| Paste mechanism | Set clipboard to chosen item, then `SendInput` synthesises Ctrl+V to the previously-foreground window. |
| Theming | Inherits the canonical `ThemeApplier.ApplyTo` + `TrayTheme.Current.Changed` pattern from Phase 29. No new theme surface. |
| Startup | Same `StartupOptions` (`--startup`, `--delay=N`) and `RunKeyStartupRegistration` as SoundTracker / ProgramHider. |

### New shared type

`shared/WindowsTrayCore/HotkeyRegistration.cs`:

```csharp
namespace WindowsTrayCore;

[Flags]
public enum HotkeyModifiers
{
    None    = 0,
    Alt     = 1,
    Control = 2,
    Shift   = 4,
    Win     = 8,
}

/// <summary>
/// Global hotkey registration wrapper. Owns a hidden NativeWindow that
/// receives WM_HOTKEY; raises <see cref="Pressed"/> with the registered id
/// on the application message-pump thread. Same NativeWindow pattern used
/// by TrayIcon and TrayTheme.
/// </summary>
public sealed class HotkeyRegistration : IDisposable
{
    public HotkeyRegistration();
    public bool Register(int id, HotkeyModifiers modifiers, Keys key);
    public bool Unregister(int id);
    public event EventHandler<int>? Pressed;
    public void Dispose();
}
```

Underlying Win32 surface added to `shared/WindowsTrayCore/Native/TrayNativeMethods.cs`:

```csharp
[DllImport("user32.dll", SetLastError = true)]
internal static extern bool RegisterHotKey(IntPtr hWnd, int id, uint fsModifiers, uint vk);

[DllImport("user32.dll", SetLastError = true)]
internal static extern bool UnregisterHotKey(IntPtr hWnd, int id);

internal const int WM_HOTKEY = 0x0312;
```

## ClipTray runtime data flow

### Capture path

```
ClipboardListener (hidden HWND, WM_CLIPBOARDUPDATE)
   |
   v
ShouldCapture? (privacy filter, evaluated in order)
   1. Paused? -> drop
   2. Session locked? -> drop
   3. Skip-sync marker present? -> drop silently
   4. Foreground process in blocklist? -> drop
   |
   v (passed filter)
ExtractContent
   - if CF_UNICODETEXT or CF_TEXT: read string, normalise CRLF, SHA-256 the UTF-8 bytes
   - elif CF_DIB / CF_BITMAP: read bitmap, encode to PNG bytes, SHA-256 the PNG bytes
   - else: drop
   |
   v
PasswordHeuristic.Evaluate (text only)
   - if matches: set IsSensitive = true (still captured, but flagged)
   |
   v
ClipboardHistory.Add (in-process)
   - dedup by hash: if existing item with same hash, move to front + update CapturedUtc
   - else: insert at front, evict from tail to enforce caps
   - pinned items are exempt from eviction
   - persist to index.json (atomic write via JsonSettingsStore<T> pattern)
   - if Image: write items\<hash>.png if not present
```

### Recall path

```
HotkeyRegistration.Pressed (default Ctrl+Shift+V)
   |
   v
PickerForm.ShowAtCursor
   - records GetForegroundWindow() so paste can target it
   - shows borderless form near cursor, focuses search box
   - loads pinned items first, then recent text, then recent images
   |
   v
User filters / navigates
   - typing: case-insensitive Contains over text items, OCR alt-text or filename over images
   - "/regex" prefix switches to regex mode
   - Up/Down: navigate
   - Ctrl+P: toggle pin on highlighted item
   - Del: delete highlighted item
   |
   v
User confirms (Enter or Click)
   - if IsSensitive: prompt "Reveal and paste? [Y/n]" inline, not a dialog
   - on confirm: ClipboardWriter.Paste(item, targetHwnd)
   |
   v
ClipboardWriter.Paste
   - Sets clipboard to item's content (CF_UNICODETEXT or CF_DIB)
   - Restores focus to targetHwnd via SetForegroundWindow
   - SendInput Ctrl-down, V-down, V-up, Ctrl-up
   - PickerForm.Close
```

### Disk quota

`AppSettings.DiskQuotaMb` (default 100). On every image capture, `ImageStore` sums sidecar sizes; if the total would exceed the cap, evict oldest non-pinned images until under the cap. Caps live in MB to keep the setting human-readable.

### Hash + dedup

SHA-256 over canonical bytes:
- Text: UTF-8 bytes of the string after CRLF -> LF normalisation
- Image: raw PNG bytes (the same bytes that go into the sidecar)

Re-copying an identical item updates `CapturedUtc` (so it floats to the top of recency-sorted views) without duplicating storage. The sidecar PNG for an image is checked for existence before write to avoid redundant disk I/O.

## Privacy implementation

The four layers, evaluated in order. Any layer can short-circuit the capture path; subsequent layers are not consulted.

### Layer 1: Pause toggle (manual)

Tray-menu item `Pause capture` (checkbox). Persisted in `AppSettings.PauseCapture`. Restored on startup. Optional hotkey to toggle (default unbound; user can bind in Settings).

When paused: capture path drops every update immediately. Tray icon overlay changes to a "paused" glyph for visibility.

### Layer 2: Session lock detection

`SessionLockMonitor` registers via `WTSRegisterSessionNotification(hwnd, NOTIFY_FOR_THIS_SESSION)` (P/Invoke into `wtsapi32.dll`). Listens for `WM_WTSSESSION_CHANGE` with `wParam == WTS_SESSION_LOCK` (sets internal `_locked = true`) and `WTS_SESSION_UNLOCK` (clears). Capture path checks the flag.

Default: enabled. Disable via `AppSettings.PauseOnLockScreen = false`.

### Layer 3: Skip-sync markers

Two Microsoft-defined clipboard formats:
- `CanIncludeInClipboardHistory` (Win10 1809+): if present and contains a single byte 0x00, do not capture
- `ExcludeClipboardContentFromMonitorProcessing`: if present at all (any value), do not capture

Implementation: `RegisterClipboardFormat` returns the format ID on first call; `IsClipboardFormatAvailable` checks presence; `GetClipboardData` reads the data byte for `CanIncludeInClipboardHistory`.

Default: enabled. Cannot be disabled in settings; this respects the application-level contract password managers expect.

### Layer 4: Foreground process blocklist

`ForegroundProcessProbe.GetCurrentName()`:
1. `GetForegroundWindow()` -> HWND
2. `GetWindowThreadProcessId(hwnd, out uint pid)` -> PID
3. `Process.GetProcessById((int)pid).ProcessName` -> string (case-insensitive)

Compared against `AppSettings.ForegroundBlocklist` (a `List<string>`).

Default seed list: `keepass`, `keepass2`, `keepassxc`, `1password`, `bitwarden`, `lastpass`, `dashlane`. Stored as ProcessName-style strings (no `.exe` suffix), matching what `Process.ProcessName` returns. User can add / remove via Settings.

The probe is best-effort: if `Process.GetProcessById` throws (process exited between calls), capture proceeds. This is a privacy fail-open, deliberately, because the password-manager case is more often "the password manager is closed and you're copying ordinary text" than "the password manager just exited mid-copy."

### Layer 5: Password heuristic (post-capture flag)

Runs only on text items, only after the four drop-layers pass. Does not drop the item; sets `IsSensitive = true`.

```csharp
public static bool LooksLikeSecret(string text, int minLen = 8, int maxLen = 64)
{
    if (text.Length < minLen || text.Length > maxLen) return false;
    if (text.Any(char.IsWhiteSpace)) return false;

    int classes = 0;
    if (text.Any(char.IsLetter))            classes++;
    if (text.Any(char.IsDigit))             classes++;
    if (text.Any(c => !char.IsLetterOrDigit(c))) classes++;

    return classes >= 2;
}
```

Settings expose `PasswordHeuristicEnabled` (default true), `PasswordHeuristicMinLength` (default 8), `PasswordHeuristicMaxLength` (default 64).

Sensitive items show in the picker as `••••••••` placeholders. Selecting one shows an inline "Reveal and paste?" confirmation (no separate dialog; same row, two buttons). Confirming reveals the value and pastes; cancelling closes the picker without revealing.

  - Type: filter (incremental, no debouncing needed at this scale)
  - Up/Down: navigate

  - Esc: close without action
  - Ctrl+P: toggle pin on highlighted

## Failure modes

| Mode | Detection | Behaviour |
|---|---|---|
| `OpenClipboard` race (another app holds it) | `IsClipboardFormatAvailable` returns false unexpectedly OR `OpenClipboard` fails | Retry 3 times at 50 ms intervals; if still locked, skip this update silently and log `clip.read.locked` |
| Bitmap too large for budget | sidecar size > `MaxImageSizeMb` (computed from quota and image cap) | Reject; log `clip.image.rejected.size`; history unchanged |
| Sidecar file write fails | exception from `File.WriteAllBytesAsync` | Add to history with `ImagePath = null` and `Kind = Image`; log warning; picker shows item with "(image unavailable)" subtitle |
| `RegisterHotKey` returns false | win32 error 1409 (ERROR_HOTKEY_ALREADY_REGISTERED) | Settings form shows inline error next to the hotkey field; app still runs; picker still accessible via tray menu |
| `index.json` corrupt on load | `JsonSettingsStore<T>` quarantine path triggers | Corrupted file moved to `index.corrupt-{timestamp}.json`; history starts empty; user notified via balloon |
| `SendInput` paste fails | non-zero return from `SendInput` API | Clipboard remains set to the chosen item; balloon "Couldn't auto-paste. Press Ctrl+V to paste manually." |
| Password heuristic false positive | user complains via flagging behaviour | They tweak settings (min/max length) or disable the heuristic entirely; existing flagged items can be unflagged via picker context menu (`Mark not sensitive`) |
| Foreground probe fails (process exited) | `ArgumentException` from `Process.GetProcessById` | Treat as not-in-blocklist; capture proceeds. Deliberate fail-open. |
| Lock screen pause flag stuck on (WTSSESSION events missed) | `_locked = true` persists into an unlocked session | "Pause toggle on lock screen" menu item lets user manually clear. Tray icon overlay surfaces the state. |

## ProgramHider migration

ProgramHider's local hotkey-registration class is deleted. Its existing call sites (typically `ProgramHiderContext` constructor + Dispose) switch to:

```csharp
_hotkey = new WindowsTrayCore.HotkeyRegistration();
_hotkey.Pressed += OnHotkeyPressed;
_hotkey.Register(id: 1, modifiers: _settings.HotkeyModifiers, key: _settings.HotkeyVk);
```

The persisted ProgramHider settings (`HotkeyModifiers`, `HotkeyVk` or whatever the existing names are) keep their types and defaults; only the registration mechanism changes.

The migration ships in the SAME phase as ClipTray so the shared library has at least two real consumers from day one. ProgramHider's tests adjust if any of them depend on the local class shape; otherwise unchanged.

## Open caveats

1. **Hotkey conflicts.** `RegisterHotKey` is exclusive; if another app (or Windows itself) has already taken Ctrl+Shift+V, ClipTray's registration fails silently. The Settings form surfaces the failure inline; the tray-menu "Show picker" item remains the fallback. Users running clipboard-history utilities side by side will need to pick non-overlapping hotkeys.

2. **`SendInput` and elevated foreground windows.** UAC integrity-level enforcement means an unelevated ClipTray cannot `SendInput` Ctrl+V into an elevated foreground window. The clipboard is still set correctly; the user pastes manually. ClipTray detects this case (paste returns 0) and surfaces a balloon. Running ClipTray elevated to fix this isn't recommended; the other four apps run unelevated and ClipTray follows that convention.

3. **Password heuristic false positives and false negatives.** Length + charset + no-whitespace catches most passwords but misses memorable / passphrase-style secrets ("correct horse battery staple") and flags some non-secrets (commit SHAs, API URLs without paths). Users tune via min/max length or disable the heuristic. The "Mark not sensitive" picker context-menu item handles per-item corrections.

4. **CF_DIB lossy interpretation.** Some legacy apps put bitmaps on the clipboard in CF_DIB v3 format with negative height + bit-fields encoded weirdly. The PNG encoder may misinterpret these. Acceptable failure mode: the image item is captured but renders as a black rectangle in the picker. Re-copy via a modern app fixes it.

5. **Image disk usage.** A single high-DPI screenshot can be 3 to 8 MB. The default 100 MB disk quota covers roughly 15 to 30 captured images, plus all 50 text items combined. Users with intense screenshot workflows should raise the quota.

6. **Concurrent clipboard read race.** Office and Citrix-style apps can hold the clipboard open for hundreds of milliseconds. The 3-retry, 50-ms backoff covers most cases; rare drops will be logged but not surfaced to the user.

7. **Telemetry / external network calls.** None, by convention with the other four apps and with the broader project rules. The `UpdateChecker` polls GitHub releases (same as every other app); that's the only external call.

8. **Future shared `WindowsClipboardCore`.** If another clipboard-aware utility ever joins the suite, the `ClipboardListener` / `ClipboardWriter` types are good candidates for promotion to a new shared lib. Phase 30 does not anticipate this; the types live in `apps/ClipTray/ClipTray/` until proven otherwise.
